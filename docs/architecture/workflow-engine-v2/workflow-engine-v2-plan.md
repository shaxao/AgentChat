# 工作流引擎 V2 改造计划

## 目标

将当前固定脚本式工作流升级为“工具执行 + AI 驱动编排”的双模式工作流引擎，支持文件/图片/音频输入、自定义工具真实执行、长时任务进度反馈、执行结果沉淀到记忆与工作文件。

## 设计约束

- 不滥用 emoji 表情作为功能图标；产品图标优先使用专业图标库，必要时使用 AI 生成的专业图标资产。
- 保持现有工作流兼容，不直接推翻已有 DSL、执行记录、模板能力。
- 大文件不得直接塞入 JSON 或模型上下文；统一通过文件资产引用、分段处理、结果合并。
- AI 驱动模式默认只能调用受控工具，不能自由访问宿主文件系统或任意网络资源。
- 每一阶段完成后必须进行代码审查，再进入下一阶段。
- 每个阶段都要明确输入、输出、权限、异常、超时和回滚边界。

## 当前问题

1. 文件交互缺失：工作流不支持上传图片和文件，实际业务覆盖不足。
2. 自定义工具空心化：当前自定义工具更像元数据，缺少真实实现逻辑和调用闭环。
3. 流程固化：当前执行更接近固定 DAG/脚本顺序，缺少 AI 根据上下文动态选择下一步的能力。

## 双模式架构

### 模式 A：普通工具模式

适用于流程明确、步骤固定、需要稳定复现的业务。

- 节点绑定确定性实现：平台内置工具、Python 脚本、HTTP API。
- 节点输入输出按 schema 传递。
- 后端按 DSL/DAG 顺序执行。
- 每个节点必须记录输入、输出、状态、耗时、错误。

### 模式 B：AI 驱动模式

适用于流程不确定、需要智能判断、需要根据中间结果调整步骤的业务。

- 流程控制权交给 Agent。
- Agent 可选择技能、工具、Python 脚本，并构造输入与解析输出。
- 根据中间结果动态决定下一步。
- 最终汇总结果返回用户。
- 工具范围必须受权限和沙箱限制，优先限定为平台内置工具与 Python 沙箱工具。

## 原生零代码工具

### AI 对话

- 节点配置选择模型、Prompt 模板、参数映射。
- 支持平台已有多模型接入。
- 输入输出 schema 固定，便于串联。

### 文件/图片上传与识别

- 支持在工作流节点输入口拖拽文件/图片。
- 上传后生成统一 artifact 引用。
- 自动调用 OCR/视觉理解能力。
- 输出统一为结构化识别结果、文本摘要、原始文件引用。

### 音频上传与转写

- 支持拖拽音频文件到节点。
- 复用后端 ASR 能力。
- 输出统一为转写文本、时间戳片段、原始音频引用。

## 后端与基础设施要求

- 单文件上限暂定 1GB。
- 大文件上传采用分片上传，支持断点续传。
- 检查 Nginx `client_max_body_size`、`proxy_read_timeout` 等配置。
- 检查后端文件处理接口，避免大文件进入内存造成 OOM。
- 超大文件内容必须分段送入模型，分段结果最终合并。
- 长时执行需要支持 SSE 或等价长连接返回中间状态。
- 执行支持超时、取消、失败重试和断点续执行。

## 阶段计划

### P0：现状审计与兼容边界

目标：明确当前系统能复用什么、需要新增什么、不能破坏什么。

审计范围：

- 工作流 DSL、节点模型、执行记录。
- 工作流画布与节点配置 UI。
- 文件上传、文件存储、工作文件、记忆文件。
- ASR/语音识别能力入口。
- 自定义工具数据模型与调用链路。
- 场景、技能、工作流、记忆之间的现有关联。

交付物：

- P0 审计结论。
- 数据模型改造建议。
- API 改造边界。
- 第一阶段开发任务清单。

审查点：

- 是否兼容已有工作流。
- 是否避免大文件进入 JSON/内存。
- 是否明确 AI 驱动模式的权限边界。

### P1：文件资产底座

目标：建立统一 artifact 模型，让工作流支持图片、音频、文档、表格等输入输出。

边界：

- 新增或复用文件资产表。
- 工作流输入输出引用 artifact id。
- 普通上传先兼容现有能力；分片上传作为独立子阶段。
- 工作文件和记忆文件能关联 workflow execution。

审查点：

- 文件大小、类型、权限校验。
- 大文件不整体读入内存。
- 工作流执行记录能追踪 artifact 来源。

### P2：原生工具节点

目标：实现 AI 对话、文件/图片识别、音频转写三类零代码工具节点。

边界：

- 固定输入输出 schema。
- 前端画布能配置和串联。
- 后端统一执行接口。
- 失败状态和错误信息标准化。

审查点：

- schema 是否稳定。
- 节点配置是否足够简单。
- 输出是否能被后续节点消费。

### P3：自定义工具实心化

目标：让自定义工具具备真实执行能力。

边界：

- 支持 Python 脚本、HTTP API、平台内置工具包装。
- 每个工具必须有参数 schema、返回 schema、权限、超时、日志。
- Python 工具必须走沙箱执行。
- 提供测试运行入口。

审查点：

- 沙箱隔离。
- 敏感信息保护。
- 日志可审计。

### P4：AI 驱动执行引擎

目标：实现 Agent 控制流程的执行模式。

边界：

- 工作流可选择普通模式或 AI 驱动模式。
- AI 驱动模式能读取上下文、选择工具、执行步骤、汇总结果。
- 工具调用范围受工作流配置和用户权限限制。
- 每次决策、调用、结果都要落执行事件。

审查点：

- 是否可解释、可追踪、可暂停。
- 是否避免无限循环。
- 是否能处理失败和重试。

### P5：长时任务进度反馈

目标：让用户实时看到工作流执行状态，而不是等待黑盒结果。

边界：

- 后端记录 run、step、event 三层执行事件。
- 前端展示执行中、当前步骤、已完成步骤、失败原因。
- 优先使用 SSE 推送进度。
- 支持取消执行。

审查点：

- 刷新页面后能恢复执行状态。
- 长任务不会导致请求超时。
- 失败原因对用户可理解。

### P6：断点续执行与阶段回放

目标：长链路失败后能从断点继续，支持执行审计与复盘。

边界：

- 保存 checkpoint：上下文、已执行步骤、artifact、模型输出、失败点。
- 支持从指定 step 继续。
- 支持查看完整执行轨迹。

审查点：

- 续执行不会重复产生副作用。
- 用户能理解恢复点。
- 审计日志完整。

## 开发流程规范

每个阶段按以下流程推进：

1. 确认阶段边界。
2. 实现最小可用闭环。
3. 补充必要测试或构建验证。
4. 进行代码审查。
5. 修复审查问题。
6. 进入下一阶段。

未经阶段审查，不进入下一阶段。
