# 三分钟项目介绍

## 开场版本

我做的是一个 AI 对话与代码开发平台，可以理解为“对话系统 + AI IDE/Agent 平台”的组合。普通用户可以进行多模型 AI 对话、文件问答、技能调用；开发场景下，用户可以提交一个代码需求，系统会自动读取项目、定位相关文件、修改代码、运行验证，并根据验证结果继续修复。

这个项目分成两个核心系统：

- Java 主系统：负责用户、登录、RBAC 权限、模型渠道、对话、记忆、技能、钱包、套餐、日志和计费。
- Python AutoCode 系统：负责代码开发任务、Agentic Loop、工具调用、工作区管理、本地 Runner、上下文索引、缓存和审批。

我重点解决的问题是：如何让 AI 不只是聊天，而是能稳定、安全、低成本地完成代码开发任务。

## 技术亮点

第一，代码开发从固定流水线升级成 Agentic Loop。

早期是“计划 -> 分组执行 -> 审查 -> 验证”的固定流程，问题是用户已经明确指出文件、函数、错误点时，系统还可能卡在流程里，甚至因为某个阶段没有文件变更而失败。后来我改成了 `observe -> decide -> act -> verify -> reconcile -> finish` 的 Agentic Loop，让 Agent 根据上下文自主决定读什么、改什么、跑什么命令、是否继续修复。

第二，做了 SystemContext Epoch。

以前 Agent 会反复读取 `.autocode` 里的记忆、计划、审查、CI 报告，token 浪费很明显。我把这些上下文抽象成 `SYSTEM_CONTEXT.json`，记录每个上下文源的路径、类型、hash、mtime、大小、优先级和摘要。每轮执行只比较 manifest diff，知道哪些上下文变了，再决定是否读取全文。

第三，做了平台级缓存体系 CacheLedger。

缓存不只是简单内存缓存，而是分层的：请求内缓存、会话缓存、工作区缓存、Provider prompt cache、长期记忆缓存、历史解决方案缓存。AutoCode 和 Java 对话系统都可以写入统一的缓存账本，记录命中率、节省 token、节省耗时、失效原因，方便后续治理成本。

第四，做了权限和本地 Runner。

工具执行分成请求批准、安全自动、替我审批等模式。高风险命令会停下来等待确认，普通命令可以自动批准。本地 Runner 则允许用户在自己电脑上运行一个小程序，通过 WebSocket 主动连接服务器，服务器发 `read_file`、`apply_patch`、`bash` 等请求，本地执行后返回结果，不需要用户开放端口，也能使用本地数据库、依赖和文件环境。

第五，做了统一安全计费和 Coding Plan 适配。

我把上游渠道成本和用户侧计费拆开。Coding Plan 只代表渠道成本模式，不代表用户免费使用。普通聊天、AutoCode、工作流统一进入计费策略，每个场景可以配置额度、钱包兜底和上游成本模式。钱包兜底默认关闭，避免支付漏洞。

## 面试收束

这个项目对我最大的锻炼是：不只是调模型接口，而是把 AI 能力工程化，包括上下文治理、工具权限、缓存命中、任务恢复、本地执行、计费安全和可观测性。我的目标是让 AI Agent 像一个可靠的工程协作者，而不是一次性问答机器人。
