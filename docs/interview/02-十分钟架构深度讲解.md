# 十分钟架构深度讲解

## 1. 总体架构

项目采用“Java 主系统 + Python AutoCode 子系统”的分层架构。

Java 主系统负责稳定业务能力：

- 用户登录、JWT、RBAC 权限
- 普通 AI 对话、SSE 流式输出
- 模型、渠道、价格、日志管理
- 记忆系统、技能系统
- 钱包、订阅、套餐、统一计费
- 给 AutoCode 暴露内部模型和用量接口

Python AutoCode 子系统负责高变化的 Agent 能力：

- 代码开发任务生命周期
- Agentic Loop
- 工具执行：搜索、读文件、写文件、命令、验证、预览
- 工作区管理
- SystemContext Epoch
- Workspace Index
- Tool Registry
- Permission Engine
- Tool Output Store
- Local Runner

这样拆分的原因是：Java 更适合承载业务系统的稳定交易链路；Python 更适合快速迭代 Agent Runtime、工具编排和本地执行能力。

## 2. AutoCode Agentic Loop

早期固定流程的问题是：任务被阶段牵着走，而不是由 Agent 根据真实上下文决策。比如用户明确说“修复 `parse_args` 中的 `args.input_file`”，系统却还提示“请描述需求”；或者冒烟测试阶段没有源码变更，就被判定失败。

现在默认进入 Agentic Loop：

```text
observe -> decide -> act -> verify -> reconcile -> finish
```

代表性入口：

```python
def _execution_mode(task: dict | None) -> str:
    configured = str((task or {}).get("execution_mode")
        or os.getenv("AUTOCODE_EXECUTION_MODE", "agentic")).strip().lower()
    return "planned" if configured in {"planned", "phase", "legacy"} else "agentic"

def _should_use_agentic_execution(task: dict | None, description: str, project_type: str = "") -> bool:
    if _execution_mode(task) == "planned":
        return False
    if task and task.get("force_planned_execution"):
        return False
    return True
```

位置：

- `agent-platform/backend/core/agent_orchestrator.py`

Agentic Loop 的行为协议是：

- observe：读取 SystemContext manifest、最近用户输入、检索计划、CI 和 Review 状态。
- decide：自己判断下一步是 search、read、edit、bash、answer 还是 ask。
- act：调用工具执行。
- verify：写入后必须运行合适验证。
- reconcile：刷新 SystemContext Epoch，根据 diff 判断是否继续。
- finish：验证通过、没有待处理用户输入、没有失败 guardrail 后才完成。

## 3. SystemContext Epoch

以前上下文管理的问题是：每次都读取 `.autocode/*.md` 全文，token 浪费，而且不知道哪些内容变了。

现在用 `.autocode/SYSTEM_CONTEXT.json` 做 manifest：

```python
SYSTEM_CONTEXT_PATH = ".autocode/SYSTEM_CONTEXT.json"

DEFAULT_SYSTEM_CONTEXT_SOURCES = (
    ".autocode/PROJECT_PROFILE.md",
    ".autocode/PROJECT_MAP.md",
    ".autocode/COMMANDS.md",
    ".autocode/PLAN.md",
    ".autocode/MEMORY.md",
    ".autocode/SESSION_SUMMARY.md",
    ".autocode/CONTEXT_SUMMARY.md",
    ".autocode/CI_REPORT.md",
    ".autocode/REVIEW.md",
    ".autocode/RETRIEVAL_PLAN.md",
    "SCRIPT_CONTRACT.md",
    "SPEC.md",
    "README.md",
)
```

每个 source 记录：

- path
- kind
- sha256
- mtime
- size
- priority
- summary
- epoch

核心收益：

- Agent 默认看 manifest，不直接拼接所有上下文全文。
- 只有 hash 变化的上下文才进入 changed sources。
- 可以发出 `system_context_indexed`、`system_context_changed`、`system_context_reconciled` 事件。
- 解决“记忆文件越来越多导致 prompt 膨胀”的问题。

位置：

- `agent-platform/backend/runtime/system_context.py`
- `agent-platform/backend/runtime/context_manager.py`

## 4. 缓存体系 CacheLedger

缓存分成六层：

| 层级 | 名称 | 作用 |
|---|---|---|
| L0 | 请求内缓存 | 单轮 Agent/对话中工具调用去重 |
| L1 | 会话缓存 | 同一 task/session 内复用上下文摘要、路由、工具结果 |
| L2 | 工作区缓存 | SystemContext、Workspace Index、文件 hash、检索计划 |
| L3 | Provider Prompt Cache | 稳定系统提示、工具定义、长期上下文排序，提高模型侧缓存命中 |
| L4 | 长期记忆缓存 | 用户偏好、项目约定、历史需求 |
| L5 | 解决方案缓存 | 错误指纹、根因、补丁摘要、验证命令、验证结果 |

统一账本字段包括：

```text
cache_layer
cache_key
tenant_id
user_id
task_id/session_id
workspace_id
epoch
input_hash
hit/miss/stale
miss_reason
token_saved_estimate
latency_saved_ms
```

AutoCode 和 Java 对话系统都可以写入同一类缓存事件。这样缓存不只是优化，而是可观测、可统计、可治理的平台能力。

位置：

- `agent-platform/backend/services/cache_ledger_service.py`
- `agent-platform/backend/api/cache.py`
- `backend/src/main/java/.../cache/CacheLedgerClient.java`

## 5. Tool Output Store

工具输出不能无限塞进 prompt。比如 `npm run build`、`ls -R`、长日志、测试失败堆栈都可能非常大。

设计是：

- 大输出写入 Tool Output Store。
- prompt 中只放摘要、输出 ID、关键错误片段。
- Agent 需要时再按 ID 读取。

好处：

- 减少 token。
- 避免活动区和聊天区被长日志刷屏。
- 错误信息仍然可追踪。

位置：

- `agent-platform/backend/runtime/tool_output_store.py`

## 6. Tool Registry

以前工具名、权限、展示文案分散在前后端，导致活动区显示 `tool_result backend glob` 这种不友好的内容。

Tool Registry 统一维护：

- 工具名称
- 中文展示名
- 功能说明
- 风险等级
- 权限策略
- 是否支持本地 Runner
- 输出展示方式

位置：

- `agent-platform/backend/runtime/tool_registry.py`

它让工具执行、权限判断、活动区展示、审批提示可以复用同一份元数据。

## 7. 权限与审批

权限系统分两层：

Java 主系统：

- RBAC 角色权限
- 管理后台权限
- 用户、套餐、钱包、模型渠道权限

AutoCode 工具层：

- 请求批准：每条命令都需要确认。
- 安全自动：低风险直接自动批准，高风险停下。
- 替我审批：非高风险 5 秒倒计时自动批准，高风险红色警告等待确认。

安全原则：

- 删除、覆盖、权限、支付、数据库迁移属于高风险。
- 高风险不能因为“自动模式”而绕过。
- 审批结果要可追踪，避免前端显示“待确认”但任务已继续。

位置：

- `agent-platform/backend/runtime/permission_engine.py`
- `agent-platform/backend/api/approvals.py`

## 8. 本地 Runner

本地 Runner 用来解决服务器环境不等于用户本地环境的问题。

设计方式：

- 用户本地启动 `autocode-local-runner.py`。
- Runner 主动连接服务器 WebSocket，不需要公网 IP。
- 服务器发送工具请求：`read_file`、`apply_patch`、`bash`、`git_diff`。
- Runner 在用户授权的项目目录内执行。
- 结果返回服务器，Agent 继续推理。

关键安全点：

- project root 限制，不能越权读写。
- `.autocodeignore` 忽略敏感和大文件。
- 权限策略仍然生效。
- 本地断连时任务不能静默切到云端破坏一致性。

位置：

- `agent-platform/backend/services/local_runner_manager.py`
- `agent-platform/backend/api/local_runner.py`
- `agent-platform/local-runner/autocode-local-runner.py`

## 9. 记忆系统

记忆不是简单把历史对话拼接进 prompt，而是分层：

- 用户偏好
- 会话摘要
- 项目记忆
- 技能记忆
- 工作区上下文
- 历史问题解决方案

Java 对话系统侧负责用户级、会话级、技能级记忆；AutoCode 侧负责项目级、任务级、工作区级记忆。

位置：

- `backend/src/main/java/com/aiplatform/backend/service/MemoryService.java`
- `agent-platform/backend/runtime/system_context.py`

## 10. 技能系统

技能系统类似插件市场，用来扩展 AI 的能力边界。

能力包括：

- 导入技能
- 审核技能
- 发布技能
- 下载技能
- 在对话和 Agent 中按需注入技能说明

设计思想是：模型不可能内置所有专业流程，所以用技能包把领域知识、操作规范和工具说明结构化。

位置：

- `backend/src/main/java/com/aiplatform/backend/controller/SkillImportController.java`

## 11. 统一计费与 Coding Plan

关键设计是把“上游渠道成本”和“用户侧计费权限”拆开。

Coding Plan 是渠道能力或成本模式，不等于用户免费使用。

统一场景：

- chat
- autocode
- workflow

每个场景可以配置：

```json
{
  "billing": {
    "scenes": {
      "chat": {
        "enabled": true,
        "costLimit": 100,
        "walletFallbackEnabled": false,
        "walletFallbackMonthlyLimit": 0,
        "upstreamBillingMode": "metered"
      },
      "autocode": {
        "enabled": true,
        "costLimit": 300,
        "walletFallbackEnabled": false,
        "upstreamBillingMode": "coding_plan"
      }
    }
  }
}
```

安全规则：

- 钱包兜底默认关闭。
- 开启钱包兜底必须有月度上限。
- 场景额度和钱包不能双扣。
- 上游 quota 错误不能扣用户钱包。
- 扣费失败不能写成功流水。

位置：

- `backend/src/main/java/com/aiplatform/backend/billing/`
- `backend/src/main/java/com/aiplatform/backend/service/UsageTrackingService.java`
- `backend/src/main/java/com/aiplatform/backend/controller/AdminController.java`
