# 高频面试问答

## 1. 这个项目和普通聊天机器人有什么区别？

普通聊天机器人主要是输入问题、输出文本。我的项目更接近 AI Agent 平台，除了对话，还能操作真实项目文件、运行命令、读取构建结果、修改代码并验证。它的难点不只是调用模型，而是把模型、安全权限、上下文、缓存、工具执行、任务恢复和计费整合成稳定系统。

## 2. 为什么要从固定流水线改成 Agentic Loop？

固定流水线适合流程明确的任务，但代码开发任务经常是动态的。比如用户已经指出具体函数或错误，系统就应该直接读文件和修改，而不是继续走“计划、分组、审查”的硬流程。Agentic Loop 让模型每轮根据上下文自己决定下一步是搜索、读文件、编辑、运行验证还是回答用户。阶段仍然保留，但只作为 guardrail，不再主导流程。

## 3. Agentic Loop 怎么避免乱改？

我用了几层约束：

- SystemContext 提供结构化上下文，避免 Agent 盲目读全项目。
- Tool Registry 标明工具风险和用途。
- Permission Engine 对 shell、文件写入、删除等高风险操作做审批。
- 写文件后必须运行验证。
- Tool Output Store 保存完整输出，prompt 只放摘要和关键错误。
- CI、Review、任务状态作为 guardrail，验证失败会继续修，不直接宣布成功。

## 4. SystemContext Epoch 解决什么问题？

它解决的是上下文成本和一致性问题。过去每轮都拼接 `.autocode` 记忆、计划、审查、CI 报告全文，token 浪费且容易混乱。现在每个上下文源都有 hash、mtime、size、summary 和 priority。Agent 默认只看 manifest，发现某个源变化后再决定读全文。这样上下文可 diff、可 reconcile、可追踪。

## 5. CacheLedger 和普通缓存有什么区别？

普通缓存只关心 key-value 命中。CacheLedger 更像缓存账本，会记录缓存层级、命中状态、miss 原因、用户、租户、任务、workspace、epoch、节省 token 和节省延迟。它的目标不是只让某个函数快一点，而是让 AutoCode 和 Java 对话系统共享一套缓存治理能力。

## 6. 为什么要做 Tool Output Store？

因为工具输出可能非常长，比如构建日志、测试失败、目录扫描。如果全部塞进 prompt，会浪费 token，还会干扰模型。Tool Output Store 把完整输出落盘或存储，prompt 里只保留摘要、关键错误和引用 ID。Agent 需要细节时再取。

## 7. 本地 Runner 为什么用 WebSocket 主动连接？

因为用户本地机器通常没有公网 IP，也不应该开放端口。Runner 主动连接服务器 WebSocket，服务器只通过这条连接下发工具请求。这样可以使用用户本地真实项目、数据库、依赖和文件环境，同时安全边界仍由授权项目目录和权限策略控制。

## 8. 本地 Runner 断开时怎么处理？

核心原则是不能悄悄切换到云端破坏一致性。如果是本地导入项目，Runner 断开后工具调用会返回 `LOCAL_RUNNER_UNAVAILABLE`，Agent 应该提示用户恢复连接，而不是在服务器空 workspace 里继续改。后续优化方向是更稳定的心跳、断点队列、请求幂等 ID 和重放机制。

## 9. 权限审批怎么设计？

我把权限分成业务权限和工具权限。业务权限在 Java RBAC 中处理；工具权限在 AutoCode Runtime 中处理。工具模式包括请求批准、安全自动、替我审批。请求批准每条都问；安全自动允许低风险自动通过，高风险停下；替我审批对非高风险做 5 秒倒计时自动批准，高风险必须人工确认。

## 10. 如何避免支付和计费漏洞？

我把上游渠道成本和用户侧计费拆开。Coding Plan 只是上游渠道模式，不代表用户免费。用户是否能用取决于 chat/autocode/workflow 场景额度、套餐和钱包策略。钱包兜底默认关闭，开启必须配置月度上限，扣款走原子钱包服务。上游 quota 错误不会扣用户钱包，扣费失败也不会写成功流水。

## 11. 为什么 Java 和 Python 两套后端？

Java 适合稳定业务域，比如用户、权限、套餐、钱包、日志、模型配置。Python 适合 Agent 工具链、文件操作、异步 WebSocket、本地 Runner 和快速迭代。两者通过内部 API 和统一缓存、计费事件衔接，既保持业务稳定，又方便 Agent 能力快速演进。

## 12. 你遇到过最典型的问题是什么？

一个典型问题是固定流程导致“没有文件变更”就失败。实际上冒烟测试或使用说明阶段本来可能不改源码。解决方式是把“是否有文件变更”从硬性成功条件降级为 guardrail，让 Agent 根据任务目标和验证结果判断是否完成。

另一个问题是上下文和日志太长。解决方式是 SystemContext manifest 加 Tool Output Store，只把摘要和索引放进 prompt。

## 13. 如果继续优化，你会做什么？

我会继续优化四个方向：

- 更稳定的本地 Runner：心跳、断线恢复、请求重放、并发限流。
- 更强的缓存命中：历史错误指纹、补丁模板、Provider prompt cache。
- 更细的权限模型：命令 AST 风险识别、路径级权限、数据库迁移审批。
- 更好的可观测性：Agent 每轮决策、工具输出、缓存命中、计费扣款都能统一追踪。
