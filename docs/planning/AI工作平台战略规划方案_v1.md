# AI 工作平台战略规划方案

**版本：v1.0**  
**日期：2026年6月27日**  
**定位：面向所有人的 AI 工作伙伴——会记住你、越用越懂你、社区共建**

---

## 一、战略愿景与核心定位

### 1.1 愿景

> **让每一个普通人都拥有自己的 AI 工作伙伴——它了解你、适应你、为你持续进化。**

不是做一个"AI 工具"，而是做一个**AI 时代的个人操作系统**。

类比：
- 微信让所有人都有社交网络
- 支付宝让所有人都有支付能力
- **我们要让所有人都有自己的 AI 工作伙伴**

### 1.2 核心定位

| 维度 | 定义 |
|------|------|
| **用户** | 所有职业的普通人，不只是开发者 |
| **价值** | 不是"帮你聊天"，而是"帮你干活" |
| **差异化** | 记忆系统（越用越懂你）+ 社区生态（各职业都有专属方案） |
| **壁垒** | 用户数据积累 + 技能生态网络效应 |

### 1.3 我们不做什么

| 不做 | 原因 |
|------|------|
| 通用聊天机器人 | 豆包/Kimi 已经做了，不需要第二个 |
| 纯开发者工具 | 市场太窄，不符合"面向所有人"的愿景 |
| 纯内容生成 | 没有持久价值，无法形成用户粘性 |

### 1.4 我们是什么

**一句话定义：** 一个会记住你的 AI 工作平台，每个人都能在这里找到或创建适合自己职业的工作方案，并通过社区共建让方案持续进化。

├─────────────────────────────────────────────────────┤

### 2.2 场景方案层——核心差异化

**这是你的平台区别于所有竞品的关键。** 不是让用户面对"对话/技能/代码"这些技术概念，而是面对**自己的职业场景**。

#### 场景方案结构

```
一个场景方案 = 
  场景入口页（行业定制界面）
  + 预置工作流模板（开箱即用）
  + 行业技能包（专业工具）
  + 记忆上下文（行业术语/业务规则）
  + 社区共建内容（用户贡献的方案）
```

#### 首批重点场景（建议）

| 场景 | 目标用户 | 核心功能 | 市场验证 |
|------|---------|---------|---------|
| 🍽️ 餐饮管理 | 餐厅老板/店长 | 排班、订货、台账、日报、成本分析 | 你自己就是用户，已验证 |
| 💻 代码开发 | 开发者/技术团队 | AI编码、调试、部署、项目管理 | 现有能力，直接复用 |
| 📊 数据办公 | 白领/管理层 | Excel分析、报表生成、PPT、数据可视化 | 高频刚需 |
| 📱 自媒体运营 | 博主/新媒体 | 内容创作、选题追踪、数据分析、排期 | 市场大、传播快 |
| 📋 行政助理 | 所有职场人 | 邮件处理、日程管理、会议纪要、待办 | Lindy已验证模式 |

### 2.3 记忆引擎——用户粘性的核心

#### 三层记忆架构

```
┌─────────────────────────────────────┐
│          长期记忆（LTM）              │
│  用户画像 · 职业身份 · 偏好习惯       │
│  业务知识库 · 行业术语               │
│  越积越厚的个人知识图谱              │
│  存储：向量数据库 + 知识图谱          │
├─────────────────────────────────────┤
│          中期记忆（MTM）              │
│  项目状态 · 近期决策 · 工作上下文     │
│  常用工作流 · 协作关系               │
│  自动摘要压缩，保留关键信息          │
│  存储：结构化数据库 + 摘要文件        │
├─────────────────────────────────────┤
│          短期记忆（STM）              │
│  当前会话上下文 · 实时对话历史        │
│  临时数据 · 当前任务状态             │
│  会话结束自动压缩转存到中期/长期      │
│  存储：内存 + 对话历史               │
└─────────────────────────────────────┘
```

#### 记忆如何让用户离不开

| 使用时间 | 记忆深度 | 用户体验 |
|---------|---------|---------|
| 第1天 | 只知道名字和职业 | "你是做什么的？" |
| 第1周 | 了解工作习惯和常用工具 | 开始推荐合适的工作流 |
| 第1月 | 积累业务数据和决策偏好 | "上次你订了这批货，这次要不要调整？" |
| 第3月 | 形成完整知识图谱 | 几乎不需要解释背景，直接干活 |
| 第6月+ | 成为不可替代的工作伙伴 | 迁移成本极高，因为所有积累都在这里 |

#### 记忆系统的技术选型

| 组件 | 推荐方案 | 理由 |
|------|---------|------|
| 向量存储 | pgvector 或 Milvus | 语义检索，支持大规模 |
| 知识图谱 | Neo4j 或 NebulaGraph | 关系追踪，实体关联 |
| 结构化存储 | PostgreSQL | 用户画像、偏好、业务数据 |
| 记忆提取 | LLM 自动提取 + 规则引擎 | 对话中自动识别值得记忆的信息 |
| 记忆检索 | 混合检索（向量+关键词+图遍历） | 多维度召回 |
| 记忆压缩 | LLM 摘要 + 时间衰减 | 模拟人类遗忘曲线 |

### 2.4 工作流引擎——让每个人都能定制

#### 工作流的三种创建方式（由简到难）

```
方式1：自然语言创建（零门槛）
─────────────────────────
用户说："每天早上帮我检查库存，低于安全值的商品自动生成订货单"
  → AI 自动生成工作流
  → 用户确认
  → 工作流上线运行

方式2：模板市场（低门槛）
─────────────────────────
浏览模板市场 → 选择"餐饮订货管理"模板
  → 填入自己的商品信息和安全库存
  → 一键启用
  → 根据需要微调

方式3：可视化编排（专业用户）
─────────────────────────
拖拽式节点编辑器
  → 连接触发器、条件判断、执行动作
  → 类似 n8n / Make 的可视化界面
  → 支持代码节点
```

#### 工作流引擎架构

```
┌──────────────────────────────────────┐
│          工作流定义层                  │
│  YAML/JSON 格式描述工作流             │
│  支持版本管理、分享、fork             │
├──────────────────────────────────────┤
│          编排引擎                      │
│  触发器（定时/事件/手动）             │
│  节点类型：AI对话/工具调用/条件/循环   │
│  多Agent协作编排                      │
├──────────────────────────────────────┤
│          执行引擎                      │
│  任务队列（长任务异步执行）            │
│  状态机管理（暂停/恢复/重试）          │
│  沙箱隔离（安全执行代码）              │
├──────────────────────────────────────┤
│          监控与反馈                    │
│  执行日志 · 错误追踪 · 性能监控        │
│  自动重试 · 失败告警                   │
└──────────────────────────────────────┘
```

### 2.5 技能市场——社区共建的生态

#### 技能生态闭环

```
用户遇到问题 → 用AI解决 → 解决方案好用 → 打包成技能分享
    ↑                                          ↓
    ←←←←←←← 其他人使用并改进 ←←←←←←←←←←←←←←←←
```

#### 技能市场核心机制

| 机制 | 设计 | 目的 |
|------|------|------|
| 创建门槛 | 对话中一键提取 + 自然语言创建 | 人人能贡献 |
| 标准格式 | SKILL.md + 配置文件 | 统一、可发现、可组合 |
| 分发方式 | 市场搜索 + 场景推荐 + 社区分享 | 让好技能被发现 |
| 质量保障 | 社区评分 + 安全扫描 + 官方认证 | 防止低质量和恶意技能 |
| 变现激励 | 付费技能 + 平台分成（建议70-80%归创作者） | 激励优质贡献 |
| 版本管理 | 类似 npm 的版本控制 | 稳定可靠 |

#### 技能分层

| 层级 | 类型 | 示例 |
|------|------|------|
| L0 基础技能 | 通用工具 | 搜索、计算、翻译、文件处理 |
| L1 场景技能 | 特定场景 | 排班管理、台账生成、报表制作 |
| L2 行业技能 | 垂直行业 | 餐饮成本分析、电商选品、教学评估 |
| L3 复合技能 | 多技能组合 | "餐厅日常运营"（排班+订货+台账+日报） |

### 3.2 关键技术选型

| 层次 | 技术选型 | 理由 |
|------|---------|------|
| **前端** | Vue3 + TypeScript + TailwindCSS | 你已有 Vue 经验，生态成熟 |
| **移动端** | uni-app 或 React Native | 一套代码多端运行 |
| **后端** | Java Spring Boot（主）+ Python（AI 服务）| Java 做业务，Python 做 AI |
| **API 网关** | Spring Cloud Gateway 或 Kong | 路由、限流、鉴权 |
| **Agent 框架** | LangChain / LangGraph / 自研 | 编排 Agent 工作流 |
| **模型路由** | 自研路由引擎 | 你的规则引擎升级版 |
| **记忆系统** | 自研（参考 Mem0 架构）| 核心竞争力，不能依赖外部 |
| **向量数据库** | pgvector（初期）→ Milvus（规模化）| pgvector 复用 PG，后期独立 |
| **知识图谱** | Neo4j | 成熟的图数据库 |
| **消息队列** | RabbitMQ（初期）→ Kafka（规模化）| 工作流异步执行 |
| **代码沙箱** | Docker 容器隔离 | 安全运行用户代码 |
| **对象存储** | 阿里云 OSS / MinIO（私有化）| 文件、媒体存储 |
| **缓存** | Redis | 会话、热数据缓存 |

### 3.3 模型路由策略（升级版）

```
用户输入
  │
  ▼
┌─────────────────────┐
│   意图识别 + 分类     │  ← 轻量模型（DeepSeek Chat）
└──────────┬──────────┘
           │
  ┌────────┼────────┬────────┬────────┐
  ▼        ▼        ▼        ▼        ▼
 简单问答  代码生成  复杂推理  多模态   记忆操作
  │        │        │        │        │
  ▼        ▼        ▼        ▼        ▼
 DeepSeek  Claude   Claude   GPT-4o   专用模型
 Chat     Sonnet 4  Opus    /Qwen-VL
  │        │        │        │        │
  └────────┴────────┴────────┴────────┘
                    │
              ┌─────▼──────┐
              │  成本优化器  │  ← 根据用户套餐、余额、
              │  降级/缓存  │     历史成功率动态调整
              └────────────┘
```

**核心原则：**
- 简单任务用便宜模型（控制成本）
- 复杂任务用强模型（保证质量）
- 根据用户反馈动态调整路由权重
- 支持用户手动指定偏好模型

## 四、分阶段路线图

### 4.1 Phase 1：筑基（第1-3个月）

**目标：跑通核心价值闭环，验证"记忆+场景+社区"模式**

| 任务 | 具体内容 | 优先级 |
|------|---------|-------|
| 记忆系统 v1 | 短期+中期记忆，用户画像自动构建，跨会话记忆 | 🔴 |
| 场景入口 v1 | 餐饮管理 + 代码开发两个场景入口页 | 🔴 |
| 工作流引擎 v1 | 定时触发 + 简单条件 + 工具调用 | 🔴 |
| 技能市场 v1 | 技能注册/发现/安装，支持 SKILL.md 标准格式 | 🔴 |
| 移动端优化 | 响应式适配 + 底部Tab导航 | 🟡 |
| 用户系统 | 注册/登录/个人中心/使用量统计 | 🟡 |

**里程碑：** 一个餐饮老板能用自己的账号，在平台上管理排班和订货，AI 记住他的所有商品和偏好。

### 4.2 Phase 2：增长（第4-6个月）

**目标：扩展场景覆盖，开放社区共建**

| 任务 | 具体内容 | 优先级 |
|------|---------|-------|
| 记忆系统 v2 | 长期记忆 + 知识图谱 + 智能遗忘 | 🔴 |
| 场景扩展 | +数据办公 +自媒体运营 +行政助理 | 🔴 |
| 工作流引擎 v2 | 可视化编排 + 多Agent协作 + 模板市场 | 🔴 |
| 技能市场 v2 | 自然语言创建技能 + 评分系统 + 创作者分成 | 🔴 |
| 多渠道触达 | 飞书/钉钉/企微机器人接入 | 🟡 |
| 开放API | 第三方开发者接入 | 🟡 |

**里程碑：** 5个以上场景上线，技能市场有100+社区贡献技能，用户能在飞书里直接使用。

### 4.3 Phase 3：规模化（第7-12个月）

**目标：建立网络效应，形成竞争壁垒**

| 任务 | 具体内容 | 优先级 |
|------|---------|-------|
| 记忆系统 v3 | MCP协议兼容 + 跨Agent记忆共享 | 🔴 |
| 场景生态 | 用户自建场景 + 行业解决方案市场 | 🔴 |
| 工作流引擎 v3 | 自然语言生成工作流 + 自动优化 | 🟡 |

## 七、核心设计原则

### 7.1 产品原则

1. **用户不懂AI也没关系** —— 所有技术概念对用户不可见，只看到自己的业务场景
2. **第一天就有用** —— 新用户进来选职业场景，立刻有预置方案可用
3. **越用越懂你** —— 记忆系统从第一次对话就开始工作
4. **社区让它生长** —— 平台提供土壤，用户共建内容
5. **简单到不需要教程** —— 如果用户需要看说明书，说明产品还没做好

### 7.2 技术原则

1. **数据为王** —— 用户数据是最核心的资产，安全、隐私、可迁移
2. **可插拔模型** —— 不绑定任何单一模型供应商
3. **API First** —— 所有能力都可通过API调用，支持多渠道接入
4. **渐进式复杂度** —— 简单的事情简单做，复杂的事情也能做
5. **记忆即基础设施** —— 记忆不是功能，是平台底层能力
