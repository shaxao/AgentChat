## Phase 1：场景系统 + 记忆增强（第 1-2 月）

**目标：数据驱动的场景系统上线，用户能看到「这个平台是干什么的」**

### P1-1. 场景数据模型与后端 API

> **依赖：** 无（全新功能）  
> **预计改动：** 后端 ~300 行 Java + 1 张表 + 1 段 SQL

**数据库（MySQL — 你现有的库）：**

```sql
CREATE TABLE scenario (
  id            BIGINT AUTO_INCREMENT PRIMARY KEY,
  name          VARCHAR(64)  NOT NULL COMMENT '场景名称：如"餐饮管理"',
  icon          VARCHAR(16)  DEFAULT '📋' COMMENT '场景图标',
  profession    VARCHAR(64)  DEFAULT '' COMMENT '目标职业',
  description   TEXT         COMMENT '场景描述',
  system_prompt TEXT         COMMENT '注入的 system prompt',
  preset_skills JSON         COMMENT '推荐安装的技能ID列表 ["uuid1","uuid2"]',
  is_official   TINYINT(1)   DEFAULT 0 COMMENT '是否官方场景',
  creator_id    BIGINT       DEFAULT 0 COMMENT '创建者ID（0=官方）',
  usage_count   INT          DEFAULT 0 COMMENT '使用次数',
  status        VARCHAR(16)  DEFAULT 'active' COMMENT 'active/disabled',
  created_at    DATETIME     DEFAULT CURRENT_TIMESTAMP,
  updated_at    DATETIME     DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

**后端接口：**

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | /api/scenarios | 场景列表（按 profession 分组，按 usage_count 排序） |
| GET | /api/scenarios/{id} | 场景详情 |
| POST | /api/scenarios | 创建场景（社区） |
| PUT | /api/scenarios/{id} | 编辑场景 |
| DELETE | /api/scenarios/{id} | 删除场景（仅自己的） |
| POST | /api/scenarios/{id}/activate | 激活场景 → 新建对话 + 注入 system_prompt + 安装技能 |

**具体文件：**
- 新建 `backend/.../entity/Scenario.java`
- 新建 `backend/.../mapper/ScenarioMapper.java`
- 新建 `backend/.../service/ScenarioService.java`
- 新建 `backend/.../controller/ScenarioController.java`
- 新建 `deploy/migrate_scenario.sql`

### P1-3. 激活场景 → 自动配置对话

> **依赖：** P1-1 + P1-2

这是场景系统的核心价值：用户点击「餐饮管理」场景后：

```
1. 新建对话
2. 注入 system_prompt（场景专属人格 + 行业知识）
3. 自动安装场景推荐的技能（排班、台账、订货等）
4. 第一条 AI 消息："你好！我是你的餐饮管理助手，我可以帮你排班、管台账..."
5. 记忆上下文自动加载（如果用户之前用过这个场景）
```

这个流程大部分能力你已有（agent system、memory injection、skill installation），需要做的是把它们**串联起来**成一个原子操作。

## Phase 2：工作流引擎 + 社区（第 3-4 月）

**目标：用户能创建自动化任务，社区开始运转**

### P2-1. 工作流引擎 MVP（自然语言创建）

> **依赖：** Phase 1 完成  
> **预计改动：** 后端 ~500 行 Java + 前端 ~400 行 React  

**只做「方式一：自然语言创建」**——战略方案里说的三种方式，先做最高频、最零门槛的。

```
用户输入："每天早上 8 点检查库存，低于安全值的商品发消息提醒我"
         ↓
AI 解析意图 → 提取触发条件、动作、参数
         ↓
生成工作流 DSL（JSON）：
{
  "trigger": { "type": "cron", "value": "0 8 * * *" },
  "steps": [
    { "id": "check", "tool": "inventory_check", "args": {} },
    { "id": "alert", "condition": "result < safety_stock", "tool": "send_message" }
  ]
}
         ↓
用户确认 → 工作流注册到调度器 → 定时执行
```

**后端需要新建：**
| 组件 | 文件 | 说明 |
|------|------|------|
| 实体 | `Workflow.java` | 工作流定义（JSON DSL） |
| 服务 | `WorkflowService.java` | CRUD + 执行 |
| 调度器 | `WorkflowScheduler.java` | 基于 Spring TaskScheduler |
| DSL 解析 | `WorkflowParser.java` | JSON → 可执行任务 |
| 控制器 | `WorkflowController.java` | REST API |

**数据库（新建表）：**
```sql
CREATE TABLE workflow (
  id         BIGINT AUTO_INCREMENT PRIMARY KEY,
  user_id    BIGINT NOT NULL,
  name       VARCHAR(128),
  dsl        JSON COMMENT '工作流 DSL',
  cron_expr  VARCHAR(64) COMMENT '触发 cron',
  status     VARCHAR(16) DEFAULT 'paused',
  last_run   DATETIME,
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

### P2-2. 场景 + 工作流 + 记忆联调

> **依赖：** P2-1

让三个系统的数据真正打通：
- 场景「餐饮管理」自动推荐「库存检查」工作流模板
- 工作流执行结果写入记忆（"6 月 28 日库存检查：牛肉安全库存告警"）
- 下次对话时，AI 能引用工作流结果

| 功能 | 说明 |

## Phase 3：生态规模化（第 5-8 月）

**目标：建立网络效应，用户来了不想走**

### P3-1. 记忆系统 v2（知识图谱）

> **依赖：** Phase 2 完成

从纯文本记忆升级到**实体关系记忆**：

```
现状：存文本 "用户叫老王，在深圳开了家火锅店，有 12 个员工"
v2：  实体: 老王 → 拥有 → 火锅店 → 位于 → 深圳
      火锅店 → 有员工 → 12人
```

技术选型：初期用 pgvector（你已有 PostgreSQL），不需要引入 Neo4j。

### P3-2. 工作流模板市场

> **依赖：** P2-1

- 官方出 20+ 模板（覆盖主要场景）
- 用户可将自己的工作流发布为模板
- 模板支持参数化（用户填入自己的数据即可用）

### P3-3. 多 Agent 协作

> **依赖：** 现有 Agent 系统

一个工作流中多个 Agent 协作：
```
餐饮日报：
  → 台账 Agent 提取销售数据
  → 成本分析 Agent 计算毛利
  → 报表 Agent 生成 Excel
  → 通知 Agent 发送到微信群
```

### P3-4. 技能市场创作者激励

- 技能创作者分成系统
- 排行榜
- 官方认证标识

## 关键决定（需要你拍板）

| 决定 | 选项 | 我的建议 |
|------|------|---------|
| 场景广场入口 | 首页直接是场景广场 vs 对话页顶部导航 | **首页直接是场景广场**（新用户体验更好） |
| 工作流调度 | 简单 cron 定时器 vs 引入 Quartz | **简单 cron**（Phase 1-2 够用，不引入额外复杂度） |
| 移动端 | PWA vs 重新开发 | **先 PWA**（改 CSS 即可，不需要新代码） |
| 知识图谱 | 现在引入 Neo4j vs Phase 3 再说 | **Phase 3 再说**（初期 pgvector 足够） |
| 上游改动 | 保持现有 Spring Boot vs 拆微服务 | **保持**（单人开发者拆微服务是自杀） |

## 风险提醒

| 风险 | 对策 |
|------|------|
| 单人开发做不完 | 严格按照优先级，一个做完再做下一个 |
| 场景不够多 | 先出 3 个官方场景（餐饮/代码/通用），社区自然增长 |
| 工作流引擎做复杂了 | 严格遵守「只做自然语言创建」约束，不做可视化编排 |
| 前端改动太大 | 尽量在现有组件上增量修改，不重写 |
